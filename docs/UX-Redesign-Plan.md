# 运营工作台 UX 重构方案

## 一、核心问题诊断

| # | 问题 | 表现 | 影响程度 |
|---|------|------|---------|
| 1 | **15 个平铺导航** | 侧边栏 3 组 15 项，无优先级区分，卖家需要逐项查找 | 严重 |
| 2 | **功能流程断裂** | 选品 → Listing → 草稿 → 发布，流程断在 Hub 的 category/draft 空壳页 | 严重 |
| 3 | **中间状态未持久化** | 选品流程全部存在 React useState 里，刷新页面就丢 | 严重 |
| 4 | **数据孤岛** | Dashboard 的指标和 OperationsData 的指标来自相同 API 但分属两页 | 中 |
| 5 | **Agent 隔离** | Agent 和运营工作台是两个独立路由，无法在运营中随时召唤 AI | 高 |
| 6 | **无关注系统** | 退货/未读聊天/待审草稿的数量存在于 API 中，导航上看不到标记 | 高 |
| 7 | **状态盲区** | 用户不知道数据多久前同步的、是否需要手动同步 | 中 |
| 8 | **上传断裂** | Hub 文件上传 `beforeUpload={() => false}` 实际是假的 | 中 |

## 二、导航重构（15 项 → 6 项）

按卖家日常工作流重组，不再按系统模块平铺：

```
运营工作台
├── 📊 工作台 Dashboard      ← 监控 + 关注事项 + 快捷入口
├── 📦 商品 Products         ← 商品管理 + 选品上架 + 草稿 + 图片
├── 📋 订单 Orders           ← 订单 + 退货 + 发货
├── 💬 客户 Customers        ← 聊天 + 提问 + 评价
├── 💰 财务 Finance          ← 流水 + 报表 + 数据分析
└── ⚙️  工具 Tools           ← 定价 + 自动运营 + 工作流 + 模板 + 知识库
```

### 合并明细

| 新导航 | 原页面合并 | 理由 |
|--------|-----------|------|
| Products | Products + Hub + Drafts + Images | 商品全生命周期：选品→Listing→草稿→发布→管理→图片 |
| Orders | Orders + Returns | 订单售后一体化：下单→发货→退货 |
| Customers | Service(chats/questions/reviews) + Marketing(部分) | 客户触点统一 |
| Finance | Finance + Reports(报表) | 收入数据 + 报表生成在同一上下文 |
| Tools | Pricing + AutoPilot + Scheduler + Templates + KnowledgeBase | 高级功能统一入口，不干扰日常卖家 |
| System | → 合并到 Settings 页面 | 系统设置本就有独立页面 |

### 侧边栏 Badge 关注系统

每个导航项显示待处理数量，数据来源于 `/api/dashboard/summary`（已有接口）：

```
📦 商品 (5)      ← 待审草稿数
📋 订单 (3)      ← 退货待处理数
💬 客户 (12)     ← 未读消息数
```

实现方式：`OperationsPage` 每 60s 轮询 summary API，动态设置 Menu 的 `badge` 属性。

## 三、选品流程持久化（新增关键修复）

### 问题

Hub 页面（选品上架）当前所有状态在 `useState` 中：

```
输入材料 → [解析] → SPU/SKU 结果 (useState) → [生成Listing] → Listing 结果 (useState)
                                                                         ↓
                                                             刷新页面 → 全部丢失
```

### 方案：SourcingSession 分段持久化

新增 `SourcingSession` 存储层，使每一步的中间结果自动保存到后端。

### 数据结构

```json
{
  "id": "sess_a1b2c3d4",
  "shop_id": "shop-1",
  "status": "parsed",
  "materials": { "text": "...", "url": "https://..." },
  "parse_result": { "spu": { "name": "...", ... }, "skus": [...] },
  "listing_result": { "title": "...", "description": "...", "keywords": [...] },
  "category_result": { "category_id": 123, "name": "..." },
  "draft_id": "draft_xxx",
  "created_at": "2026-05-09T10:00:00",
  "updated_at": "2026-05-09T10:05:00"
}
```

### 状态流转

```
input  →  parse()  →  parsed  →  generateListing()  →  listing_generated
                                                          ↓
                                                    matchCategory()
                                                          ↓
               draft_created  ←  createDraft()  ←  category_matched
```

### 后端新增

| 文件 | 改动 |
|------|------|
| `src/icross/core/storage/ozon_data.py` | 新增 `SourcingSessionStorage` 类（基于 JsonStore） |
| `src/icross/api/routers/sourcing.py` | 新增 Router：`POST/GET /sourcing/sessions`，`PUT/DELETE /sourcing/sessions/{id}` |
| `src/icross/api/main.py` | 注册新 router |

### 前端 Hub 改造

- **自动恢复**：Hub 加载时检查当前店铺是否有未完成的 session，自动恢复进度到对应步骤
- **自动保存**：每步操作成功后调用 `PUT /api/sourcing/sessions/{id}` 保存当前状态
- **手动管理**：输入材料 tab 顶部展示"未完成选品"列表，可切换/删除/新建
- **关闭/完成清理**：创建草稿成功后标记 session 为 completed 或删除

### 用户体验流程

```
首次进入 Hub:
  ┌─ 输入材料 → 解析 → SPU/SKU 页面 ─┐
  │  (自动保存到 backend)               │
  └────────────────────────────────────┘
        ↓ 用户刷新浏览器
  ┌─ Hub 加载 → 检测到未完成 session ─┐
  │  → 自动恢复到 SPU/SKU 页面         │
  │  → 继续生成 Listing                 │
  └────────────────────────────────────┘
        ↓ 用户完成所有步骤
  ┌─ 创建草稿成功 → session 标记完成 ─┐
  │  下次进入 Hub → 新建 session       │
  └────────────────────────────────────┘
```

### 对现有选品流程的修复

除了持久化，同时修复 Hub 的断裂点：

- 文件上传从假上传改为调用 `/api/upload` 真实上传
- 类目匹配接入已有 `/api/categories/match` API（接口已实现但前端未连接）
- 生成草稿接入 `/api/drafts/create` API
- 草稿审核通过后自动关联到商品列表

## 五、Agent 集成 — 核心差异化

当前 Agent 和运营页面割裂，这是最大的 UX 损失。

**方案：运营页面右下角浮动 Agent 入口**

```
┌─ 页面内容 ──────────────────────────────────────────────┐
│                                                         │
│                                     ┌─────────────────┐ │
│                                     │ 🤖 AI 助手       │ │
│                                     │ "帮我分析这批库存"│ │
│                                     │ [输入...] [发送] │ │
│                                     └─────────────────┘ │
│                                    ↑ 右下角浮动按钮触发  │
└─────────────────────────────────────────────────────────┘
```

**关键集成点：**
- 在商品页面选中某商品后点 AI 助手 → Agent 自动获得该商品上下文
- "分析我的店铺数据" → Agent 在当前页面展示分析结果
- Agent 执行结果可以直接在当前页面呈现（如分析完直接展示报表卡片）

## 六、数据流与状态治理规范

每个页面需要显式处理 5 种状态：

| 状态 | 表现 | 当前覆盖度 |
|------|------|-----------|
| ✅ 有数据 | 正常渲染 | 全部 |
| ⏳ 加载中 | Skeleton / Spin | 大部分（DataTable 有内置） |
| 🕳️ 空状态 | 引导性提示 + 操作按钮 | 参差不齐 |
| ❌ 错误 | 错误提示 + 重试按钮 | 较差（仅 message.error） |
| 📡 数据陈旧 | "上次同步: 2小时前" + 同步按钮 | 完全缺失 |

需新增组件：
- `SyncIndicator` — 上次同步时间 + 同步按钮
- `ErrorState` — 错误提示 + 重试按钮（替代 message.error）
- `LoadingSkeleton` — 按页面结构定制的骨架屏

## 八、Phase 1 详细技术方案

### 8.1 导航精简

**文件：** `OperationsPage.tsx`

将 `sidebarSections` 从 15 项合并为 6 项，只改导航配置，页面内容保持原样可通过路由访问。删除的导航项通过 URL 直接访问或从关联页面跳转可达。

### 8.2 SourcingSession 持久化

**后端新增文件：**

| 文件 | 内容 |
|------|------|
| `src/icross/core/storage/ozon_data.py` | 新增 `SourcingSessionStorage` 类 |
| `src/icross/api/routers/sourcing.py` | `POST/GET /api/sourcing/sessions` + `PUT/DELETE /api/sourcing/sessions/{id}` |

**前端改动：** `Hub.tsx`

核心逻辑：

```
onMount:
  GET /api/sourcing/sessions?shop_id=xxx
  → 如果有未完成的 session，自动加载并跳转到对应步骤

afterParse:
  POST /api/sourcing/sessions (创建) 或 PUT (更新)
  → 保存 parse_result，设置 status="parsed"

afterGenerateListing:
  PUT /api/sourcing/sessions/{id}
  → 保存 listing_result，设置 status="listing_generated"

afterCreateDraft:
  PUT /api/sourcing/sessions/{id}
  → 保存 draft_id，设置 status="draft_created" (或删掉 session)
```

### 8.3 关注系统 Badge

**文件：** `OperationsPage.tsx`

```typescript
// 每 60s 轮询
useEffect(() => {
  const fetchSummary = async () => {
    const { data } = await axios.get('/api/dashboard/summary', { params: { shop_id: currentShop } })
    setBadges({
      products: data.pending_drafts || 0,
      orders: data.pending_returns || 0,
      customers: data.unread_chats || 0,
    })
  }
  fetchSummary()
  const timer = setInterval(fetchSummary, 60000)
  return () => clearInterval(timer)
}, [currentShop])
```

然后 map 到 Menu items 的 `badge` 属性。
